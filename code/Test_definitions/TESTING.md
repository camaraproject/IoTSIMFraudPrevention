# Testing Strategy for IoT SIM Fraud Prevention API ŌĆō Notification Delivery

This document describes the testing approach for validating that the IoT SIM Fraud Prevention Subscriptions API correctly delivers notifications when subscribed events occur. It supplements the contract tests defined in the Gherkin feature files by addressing endŌĆætoŌĆæend scenarios that involve event generation, sink behaviour, and subscription constraints.

## Overview

The Subscriptions API allows clients to create subscriptions for IMEI change and area change events. When such an event occurs, the server must send a `POST` request to the registered callback URL (`sink`) with the appropriate authentication (`sinkCredential`). The tests outlined below verify that:

- Notifications are delivered for each event type.
- Notifications cease when subscription constraints (`subscriptionExpireTime`, `subscriptionMaxEvents`) are met.
- Notifications stop after a subscription is deleted.
- The correct sink credentials are used (PLAIN, ACCESSTOKEN, REFRESHTOKEN).
- (Future) Sink modifications are honoured.

## Test Environment Requirements

To execute these tests, the following components must be available:

1. **A running instance of the API server** under test, configured to send notifications to external endpoints.
2. **A mock sink** ŌĆō a lightweight HTTP server that can receive notifications, validate headers and payloads, and return appropriate HTTP status codes. The mock sink must be able to:
   - Capture all incoming requests for later inspection.
   - Respond with configurable status codes (e.g., 204 for success, 400 for malformed requests).
   - Optionally simulate failures (e.g., 410 Gone) to test client resilience.
3. **Event simulation capability** ŌĆō the ability to trigger IMEI change or area change events for a given device. This may be achieved through:
   - Internal test hooks in the API server.
   - Network simulators that mimic device behaviour.
   - Direct database manipulation (if events are generated by state changes).
4. **Time control** ŌĆō for tests involving expiration (`subscriptionExpireTime`), the ability to advance the system time (or wait realistically) is needed. In a CI environment, time can be mocked or tests can be designed with short expiration periods.

## Test Scenarios

The detailed Gherkin scenarios are located in `iot-sim-fraud-prevention-subscriptions-notifications.feature`. Below is a summary of each scenario group:

### Basic Delivery
- **IMEI_CHANGE** ŌĆō Verify that an IMEI change triggers a notification with the correct payload.
- **AREA_CHANGE** ŌĆō Verify that an area change (enter/leave) triggers a notification with the correct payload.
- **Multiple Events** ŌĆō Ensure each event results in a separate notification.

### Subscription Constraints
- **Expiration Time** ŌĆō Confirm that no notifications are sent after the `subscriptionExpireTime` has passed.
- **Maximum Events** ŌĆō Confirm that after `subscriptionMaxEvents` notifications have been sent, further events are ignored.
- **Deletion** ŌĆō Confirm that deleting a subscription stops all future notifications.

### Sink Credentials
- **PLAIN** ŌĆō Verify that Basic authentication is used correctly.
- **ACCESSTOKEN** ŌĆō Verify that Bearer token authentication is used.
- **REFRESHTOKEN** ŌĆō Verify that the provided access token is used (refresh flow is not tested here but can be extended).

### Future / Update Scenarios (Placeholder)
- **Sink Modification** ŌĆō If the API later supports updating the sink URL, test that notifications go to the new URL.
- **Sink Nullification** ŌĆō Test that disabling the sink stops notifications.
- **Credential Update** ŌĆō Test that updated credentials are used.

## Mock Sink Implementation Example

A simple Python Flask application can serve as a mock sink:

```python
from flask import Flask, request, jsonify
import logging

app = Flask(__name__)
notifications = []

@app.route('/notify', methods=['POST'])
def notify():
    notifications.append({
        'headers': dict(request.headers),
        'body': request.get_json()
    })
    # Simulate successful reception
    return '', 204

@app.route('/reset', methods=['POST'])
def reset():
    notifications.clear()
    return '', 204

@app.route('/notifications', methods=['GET'])
def get_notifications():
    return jsonify(notifications)
